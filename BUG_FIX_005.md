## Bug 描述

**标题:** 无法让第三名玩家在3人桌成功入座

**严重程度:** 🔴 高 (阻塞多人游戏)

**测试场景:** 场景1 - 3人桌基础打牌玩法测试

**复现步骤:**
1.  创建3人桌。
2.  第一、二名玩家成功入座。
3.  第三名玩家尝试入座。

**预期结果:**
-   第三名玩家成功入座。

**实际结果:**
-   API 返回 400 Bad Request 错误，入座失败。

---

## 根本原因

在 `tables.py` 的 `sit()` 函数中，当第二名玩家入座时，`auto_start=True` 的逻辑会立即触发 `start_game()`。这导致牌桌状态 `t["status"]` 从 `"waiting"` 变为 `"playing"`。

当第三名玩家尝试入座时，会触发以下检查：

```python
# 游戏进行中不允许新玩家入座
if t["status"] != "waiting":
    return False, "游戏进行中，无法入座"
```

由于此时状态已经是 `"playing"`，第三名玩家的入座请求被拒绝。

**问题根源：** 游戏开始得太早，阻止了更多玩家加入。

---

## 修复方案

**文件:** `tables.py` - `sit()`

**修复策略：**
-   允许玩家在游戏刚开始、但**尚未有任何实际行动**时加入牌桌。
-   这为玩家提供了更大的灵活性，既可以保证2人桌自动开始，也方便后续玩家加入。

**修复代码:**
```python
# 游戏进行中不允许新玩家入座（但如果游戏刚开始且还在等待阶段，允许入座）
if t["status"] == "playing":
    # 检查游戏是否刚开始（还没有发牌）
    game_state = t.get("game_state")
    if game_state:
        stage = game_state.get("stage")
        # 如果还在 PREFLOP 且没有人行动过，允许入座
        if stage and hasattr(stage, 'name') and stage.name == "PREFLOP":
            # 检查是否有人已经行动
            has_action = any(
                p.get("last_action") for p in game_state.get("players", {}).values()
            )
            if not has_action:
                # 游戏刚开始，还没人行动，允许入座
                pass
            else:
                return False, "游戏进行中，无法入座"
        else:
            return False, "游戏进行中，无法入座"
    else:
        # 如果没有 game_state，说明游戏已深入进行，不允许入座
        return False, "游戏进行中，无法入座"
```
